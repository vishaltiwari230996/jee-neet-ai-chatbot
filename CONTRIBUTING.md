# Contributing to NeetAI

Short list of conventions. Read this before your first PR.

## Setup

```bash
pip install --user uv          # once
cp .env.example .env
make bootstrap                  # installs workspace + pre-commit
```

## Daily workflow

```bash
make dev        # start postgres + redis
make api        # FastAPI with reload
make check      # lint + types + import-linter (run before pushing)
make test       # unit tests
```

## Code conventions

* **Type hints everywhere.** mypy runs in `strict` mode in CI.
* **No SDK imports in domain packages.** Use the `LLMClient` /
  `VectorStore` / `Cache` Protocols in `neetai_ports`.
* **No business logic in `apps/api`.** Routers translate HTTP to typed
  domain calls and nothing else.
* **Pure functions where possible.** Inject `Clock` instead of calling
  `datetime.now()`.
* **No magic.** If a piece of behavior would surprise a new engineer
  reading the file, it's wrong; refactor or document.
* **Comments explain *why*, not *what*.** The code already says what.

## Module boundaries

These are enforced by `import-linter` in CI:

* `neetai_core` — depends on stdlib + pydantic only.
* `neetai_ports` — depends on `neetai_core` only.
* Domain packages — depend on `neetai_core` and `neetai_ports`.
* Adapters — depend on `neetai_core` and `neetai_ports`.
* `neetai_api` — depends on everything (it composes the system).

If you need to violate a contract, **write or update an ADR first.**

## Tests

* **Unit** — fast, hermetic, no docker. Default `make test`.
* **Integration** — marked `@pytest.mark.integration`; require local
  docker stack. Run with `make test-all`.
* **LLM** — marked `@pytest.mark.llm`; opt-in only, never in CI.

Every PR must add or update tests for the behavior it changes.

## Commits & PRs

* Conventional commits encouraged (`feat:`, `fix:`, `chore:`, `docs:`).
* PR titles describe the change a reviewer can verify in <5 min.
* Architectural changes need an ADR in `docs/adr/`.
* No PR is merged red. CI must pass.

## Secrets

* Never commit a key. Pre-commit's `gitleaks` will catch most leaks; it
  is not a substitute for being careful.
* Use `.env` locally and your platform's secret manager in deployed
  environments.

## Adding a new external dependency

1. Add an ADR explaining why.
2. Define a port in `neetai_ports` (unless trivially internal).
3. Implement the adapter under `adapters/<name>`.
4. Wire it into `apps/api/container.py`.
5. Add a fake or in-memory implementation for tests.

That's the whole list.

## Gotchas worth knowing

* **SQLAlchemy 2.0 `MappedAsDataclass` field ordering.** Our ORM models
  use the typed `Mapped[]` API in dataclass-mode. The Python dataclass
  rule applies: every column with a `default` / `default_factory`
  must come *after* every column without one. If you add a new
  non-default column, put it above the defaulted block.
* **`Mapped[]` + mutable class attributes** trip `RUF012` for
  `type_annotation_map`. Annotate it with `typing.ClassVar`.
* **Domain enums vs persistence strings.** The Postgres layer stores
  enums as `VARCHAR` with `CHECK` constraints (cheap to add new
  values without a migration). When converting in the mapper, always
  call `.value` on the way out and `Enum(value)` on the way in — never
  pass the enum instance to the ORM column directly.

## Database changes

1. Update SQLAlchemy models in `adapters/db_postgres/models.py`.
2. Update the corresponding `BankQuestion` / `Student` / etc. in
   `neetai_ports/repositories.py` if the domain shape changes.
3. Update the mappers in `adapters/db_postgres/mappers.py`.
4. Generate a migration: `make migrate-new MSG="add foo column"`.
5. Review the autogenerated diff — Alembic is good but not psychic.
6. Apply locally: `make migrate`.
7. Update the `db_fake` in-memory adapter if behaviour changed.
